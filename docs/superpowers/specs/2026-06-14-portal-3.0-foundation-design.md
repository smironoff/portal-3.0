# Portal 3.0 Foundation: Design

**Date:** 2026-06-14
**Status:** Approved (pending written-spec review)
**Sub-project:** Foundation / app shell (sub-project 1 of the Portal 2.0 to 3.0 port)

## Context

Portal 2.0 is a mature React 17 single-page application for a regulated CFD
brokerage: roughly 113k lines, 708 `.tsx` files, 412 `.module.scss` files,
and 50 or more screens. It is built on Create React App (react-scripts 4) with
craco, react-router-dom v5, Redux Toolkit + redux-saga, Formik, i18next with
Applanga, and SCSS Modules. It carries heavy integrations: Keycloak SSO plus a
custom TFBO token layer, multiple KYC providers (SumSub, TrustDock, GreenID,
Jumio, GBG), multiple payment processors (Checkout.com, Braintree, Google Pay,
PayPal), an IB/partner portal, and PAMM.

The port to a modern stack is too large for a single design and plan. It is
decomposed into independently shippable sub-projects, built in this order:

1. **Foundation / app shell** (this document).
2. Auth + onboarding/KYC vertical.
3. Dashboard + accounts vertical.
4. Funds vertical (deposit, withdraw, transfer, processors).
5. Profile/settings vertical.
6. IB/partner portal and PAMM.

Each later vertical gets its own spec, plan, and implementation cycle, and all
depend on this Foundation.

## Goal

Stand up a modern, empty-but-running application shell that every later feature
vertical builds on. No user-facing features ship in this sub-project. The
Foundation is complete when the shell, theme, routing, state architecture, auth
layer, and i18n all work end to end, validated by a single throwaway
authenticated "hello" screen.

## Decisions (confirmed)

| Area | Decision |
| --- | --- |
| Build tool | Vite + React 19 + TypeScript (strict mode) |
| Router | TanStack Router (typed routes) |
| Server state | TanStack Query |
| Client state | Zustand (small, split stores) |
| State libraries removed | Redux, Redux Toolkit, redux-saga |
| Component library | MUI v9 (theme + `sx` + `styled()`); no SCSS Modules |
| Forms | React Hook Form + Zod (replacing Formik) |
| Auth/API | Reworked typed HTTP client; built test-first; security review gated |
| i18n | Keep i18next + react-i18next + Applanga |
| Error tracking | Keep Sentry; Elastic APM deferred (optional, non-blocking) |
| Repository | Clean rebuild in the `portal-3.0` folder; legacy retained as read-only reference |
| Configuration | Build-time `.env` files per environment via Vite modes (`VITE_` vars), validated with zod. Supersedes the original runtime `/config.json` approach. |

## Architecture

### 1. Build and tooling

- **Vite + React 19 + TypeScript strict.** Replaces CRA/craco.
- **ESLint flat config + Prettier.** TypeScript strict, path aliases (for
  example `@/`).
- **Build-time environment configuration.** Configuration is supplied through
  Vite `.env` files selected by build mode (`.env.development`, `.env.staging`,
  `.env.uat`, `.env.production`), using `VITE_`-prefixed variables exposed on
  `import.meta.env`. The values are validated with zod at startup and exposed
  through a typed config provider. This produces one build per environment
  (`vite build --mode <env>`). Only public values are stored (API URLs, public
  client id, public captcha site keys, Sentry DSN); no secrets are committed.
  (This supersedes the legacy runtime `/config.json` model.)
- **Testing:** Vitest + React Testing Library for unit and component tests;
  Playwright for end-to-end smoke tests.

### 2. Routing shell (TanStack Router)

- Typed route tree mirroring the legacy top-level structure:
  - Public: `/account/*`
  - Authenticated: `/dashboard/*`, `/profile/*`, `/create-account`
  - Partner: `/partner/*`
  - Standalone provider pages kept at top level: `/checkoutcom`, `/sumsub`,
    `/trustdock`, `/greenid`, `/scanid/*`, `/payment-result`.
- Auth-guarded layout routes: when the session gate is closed, authenticated and
  partner routes redirect to login.
- Lazy-loaded route components and a route-level error boundary.
- For the Foundation, only the shell layouts plus one throwaway authenticated
  screen are wired; feature routes are stubs added in later sub-projects.

### 3. MUI design system (replaces SCSS Modules)

- **MUI v9** with a central theme built from the existing design tokens:
  - Light and dark palettes plus the IB brand variant, ported from the roughly
    170 CSS custom properties already defined in `src/css/_light.scss`,
    `_dark.scss`, and `src/ib/css`.
  - Breakpoints mapped from the legacy values: mobile 600, tablet 900,
    laptop 1024.
- **Styling via theme + `sx` prop + `styled()`** for complex cases. No
  `.module.scss` files. The only global stylesheet covers font face
  declarations and a minimal CSS reset.
- **Multi-language typography:** Figtree plus Noto Sans JP/SC/TC, Thai, and
  Arabic, selected by active language.
- **RTL support** via MUI's RTL cache (`stylis-plugin-rtl`), coordinated with
  the language/theme state.
- **Starter set of shared primitives** wrapping MUI so verticals have a stable
  internal API: Button, TextField/Input, Select/Dropdown, Checkbox,
  Dialog/Modal, Banner, PhoneInput, CountryDropdown, ThemeSwitcher. These mirror
  the partial `ui-4.0` set in the legacy app and are wired for React Hook Form.

### 4. State architecture (Redux and saga removed)

- **TanStack Query** for all server state: a configured `QueryClient` with
  conventions for stale time and retry, plus helpers for the polling flows the
  sagas previously owned (account-status polling, inactivity timeout).
- **Zustand** for genuine client state, split into small stores:
  - Session/auth gate (logged-in flag, token-derived state).
  - UI (language, theme, screen size/breakpoint, sidebar).
  - Notifications (transient toasts and banners).
- A short written guide maps each legacy saga to its new home (a query, a
  mutation, or a store effect) so the later verticals follow one consistent
  pattern. The genuinely stateful flows to account for: cold-boot
  initialisation, login/2FA continuation, inactivity timeout, application-status
  polling, and multi-step application submission.

### 5. API and auth layer (reworked, typed, test-first)

- A **typed HTTP client** wrapping `fetch`, with typed endpoint modules. If the
  backend exposes an OpenAPI/Swagger schema, types are generated from it;
  otherwise types are hand-authored from the legacy `src/utils/types.ts`.
- An **auth/token manager** covering Keycloak SSO and the TFBO session/token
  model:
  - Automatic refresh on 401 / `NOT_AUTHORIZED`, using a single-flight mutex so
    concurrent requests trigger only one refresh (as the legacy app does with
    `async-mutex`).
  - Token-expiry logout and redirect to login.
  - A clean interface so the rest of the app never touches token storage
    directly, and TanStack Query's error handling integrates with it.
- This layer is built test-first and carries the highest test coverage in the
  Foundation. **A security/compliance review is a required gate** before this
  layer is considered done, given the regulated context and that auth is the
  highest-risk area to rewrite.

### 6. i18n and observability

- **Keep i18next + react-i18next + Applanga** with runtime namespace loading,
  coordinated with the RTL and theme setup. The translation pipeline does not
  change.
- **Sentry** retained for error tracking, wired at the provider level with an
  error boundary. Elastic APM is deferred and treated as optional; it does not
  block the Foundation.

### 7. Application entry and provider composition

A single composed provider tree wraps the app, in roughly this order
(outermost to innermost): Sentry error boundary, runtime-config provider,
TanStack Query provider, MUI theme/CSS-variables provider (with RTL cache),
i18n provider, then the TanStack Router.

## Out of scope (Foundation)

Login/2FA, onboarding/KYC, dashboard, funds, profile, IB/PAMM, and all feature
screens. Each is a later sub-project. The Foundation proves the shell only.

## Definition of done

- `npm run dev` serves the app via Vite; `npm run build:<env>` produces a
  per-environment bundle from the matching `.env.<env>` file.
- Lint, type-check, unit tests (Vitest), and a Playwright smoke test all pass in
  CI-equivalent commands.
- The MUI theme renders in light, dark, and IB-brand variants, with working RTL.
- The provider tree mounts cleanly: Sentry, config, Query, theme, i18n, router.
- The auth/API layer can log a user in (against a test/UAT backend or mocked
  endpoints), refresh on 401 via single-flight, and log out on token expiry,
  with tests covering each path.
- One throwaway authenticated "hello" screen is reachable only when the session
  gate is open and confirms an authenticated API call succeeds end to end.
- The security/compliance review of the auth layer has been completed.

## Risks and notes

- **Auth rework is the primary risk.** Mitigated by test-first development, a
  single-flight refresh mutex, isolation behind a clean interface, and a
  mandatory security/compliance review gate.
- **Theme fidelity.** Porting 170-plus CSS custom properties into the MUI theme
  must preserve the existing light/dark/IB visuals; verified visually against
  the legacy app.
- **Build-time config.** Configuration is compiled per environment from `.env`
  files; the deploy pipeline must run the correct `build:<env>` and never ship a
  development build. A guard rejects a `production`/`staging` build that still
  carries development values.
- **OpenAPI availability.** Whether the backend provides a schema determines
  generated versus hand-authored types; either path is acceptable.
